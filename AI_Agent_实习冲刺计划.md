# 第四部分 · 面试准备

## 4.1 高频面试问题 + 回答策略

### Q：讲一下你这个项目的架构？
对着架构图从上往下讲——用户层 → Agent 编排（**重点讲手写 ReAct，体现你懂底层**）→ RAG / 工具 / 记忆三大能力 → 贯穿的 Eval 和可观测性。
**关键**：主动提手写 ReAct 而非直接用框架，这是你的差异化王牌。

### Q：为什么用 Agentic RAG 而不是普通 RAG？
普通 RAG 是固定检索一次就生成，遇到复杂问题容易漏信息。我加了 Router 动态选检索策略 + Critic 评估检索质量不够则重检索。
**关键**：用 RAGAS 数据证明——Context Recall 从 X 提升到 Y。数据驱动是核心。

### Q：工具调用是怎么实现的？（必问）
讲 ReAct 循环——`stop_reason` 判断 `tool_use` vs `end_turn`，工具结果用 `role:user` 的 `tool_result` 块反馈（**强调这是 Anthropic 规范，不是 OpenAI 的 `role:tool`**），讲 `MAX_ITERATIONS` 防死循环。
**关键**：这些细节证明你真的手写过，不是抄教程。

### Q：怎么评估你的系统好不好？
RAGAS 四指标——Faithfulness / Answer Relevancy 评估生成，Context Precision / Recall 评估检索。讲诊断逻辑：先看 Recall（找得到吗）→ Precision（排对了吗）→ Faithfulness（有幻觉吗）。
**关键**：有评估体系 = 工程成熟。

### Q：遇到过什么难点 / 踩过什么坑？
举真实的——比如"工具描述模糊导致 Agent 不调用工具，我做了对比实验发现描述质量直接影响调用准确率"，或"Mem0 记忆越界检索的问题，我加了相关性过滤"。
**关键**：讲真实的坑和解决过程，比讲顺利更可信。

### Q：如果让你继续优化，你会做什么？
这里提你"暂缓"没学的——Multi-Agent 拆分复杂任务、MCP 标准化工具接入、Harness Engineering 用棘轮原则固化错误、可观测性接入生产监控。
**关键**：展示你有清晰的进阶蓝图，知道行业前沿在哪。

# 附录 · 第一阶段核心机制速查（面试前复习）

这些是你手写 ReAct 时掌握的底层机制，面试必考，务必能脱口而出：

1. **`stop_reason` 的两种值**
   - `tool_use`：Claude 决定调用工具，需要执行并反馈结果
   - `end_turn`：Claude 认为信息足够，直接给出最终答案

2. **工具结果的反馈格式（Anthropic 规范）**
   - 必须用 `role:user` 消息，内含 `type:tool_result` 的 content block
   - 必须带 `tool_use_id` 与 Claude 的请求对应
   - 注意：Anthropic 没有 `role:tool`，那是 OpenAI 的格式

3. **为什么要循环所有 content blocks**
   - `response.content` 只含本轮新生成内容（天然增量）
   - 循环是为了处理同一轮里的多个并行工具请求，不能只取第一个

4. **`MAX_ITERATIONS` 防死循环**
   - 防止工具永远报错却一直重试、或 Agent 陷入循环依赖
   - 更好的做法：工具层加失败计数，连续失败 N 次返回终止信号

5. **Pydantic 的两层作用**
   - 业务数据验证（用户输入的交易记录等）
   - LLM 结构化输出的校验层和自动重试依据（配合 instructor 库）

6. **Context Engineering 四操作**
   - Write：存到上下文窗口之外（scratchpad / 记忆库）
   - Select：只检索当前任务需要的信息（RAG / 记忆检索）
   - Compress：超预算时摘要压缩（Rolling Summary）
   - Isolate：复杂任务拆给多个 Agent，各自专注小上下文
